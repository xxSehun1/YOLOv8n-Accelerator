# Critical Review of PLAN2 (Full Backbone RTL Completion)

## 1. Overall Assessment
`PLAN2.md` represents a significant step in the right direction. The adoption of the "Non-Negotiable Correctness Policy" (100% Bit-Exact matching with no tolerance) and the structured "Layer-Level Verification Ladder" (Phase 5) are excellent verification strategies. 

The strategy to align directly with the generated ISA (`full_instructions.txt`) is correct. The RTL team is not responsible for handling operations like `BatchNorm` folding or structural `Split` operations; it is assumed the Compiler team handles these by manipulating Weights/Bias and defining precise SRAM read/write address boundaries. The RTL's job is purely to execute the given ISA correctly.

However, the plan contains a major architectural shortcut that must be addressed before approval.

## 2. The Unacceptable Shortcut: "Correctness-First" ComputeTop
In Phase 1 and Phase 3, the plan proposes building a `ComputeTop` characterized by:
*   "single SRAM port for compute"
*   "sequential reads"
*   "software-like nested loops"

**Critique:** 
While building a simplistic, sequential Verilog state machine is an acceptable way to create a *temporary* RTL Golden Reference to verify the integer math logic (Quantization, SiLU, ADD), **it is unacceptable for the final project sign-off.** 

A sequential, single-port MAC engine is not an accelerator. Running the full YOLOv8n backbone (Phase 6) on such an architecture would result in absurdly long simulation times and completely fails to verify the real hardware complexity (memory bandwidth, backpressure, parallel PE array execution).

**Required Action:**
*   You may keep the "Correctness-First" `ComputeTop` strictly as a Phase 3/4 debugging milestone.
*   You **MUST** add a new phase (e.g., Phase 6.5 or modify the Definition of Done) specifying that the final Full Backbone Verification **must be executed on the real Systolic Array / PE Array architecture** integrated with the PingPong buffers. Passing the backbone on the sequential model does not count as project completion.

## 3. Feedback on Primitive Compute Blocks (Phase 2)
The breakdown into `QuantizeUnit`, `ActivationUnit`, `AddUnit`, etc., is logical and modular.

**Requirement for QuantizeUnit & AddUnit:**
When implementing the bit-exact integer math, the exact bit-widths of the internal datapath (e.g., 32-bit accumulators, bounds for the Adder Trees) and the exact overflow/underflow handling mechanisms must be explicitly defined in the RTL specifications. Do not rely on implicit Verilog truncation.

## 4. Conclusion and Required Updates to PLAN2
To gain approval for PLAN2, the following changes must be made:

1.  **Acknowledge Compiler Responsibility:** Explicitly state that operations like `Split` and `BatchNorm` are handled by the compiler's memory mapping and weight generation, and the RTL will strictly execute the `CONV`/`POOL`/`ADD` commands as provided in the ISA.
2.  **Mandate Real Hardware Sign-off:** Update the "Definition of Done" and Phase 6 to explicitly state that the final Bit-Exact P3/P4/P5 outputs must be generated by simulating the **full parallel Systolic Array datapath**, not the "Correctness-First" sequential loop engine.

Once these conditions are integrated into the plan, you are clear to proceed with Phase 0 (Freezing the Golden Contract).
